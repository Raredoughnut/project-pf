## 6. 데이터 패칭 전략 - 서버 컴포넌트 vs 비동기 상태관리(TanStack Query)

데이터의 **성격**에 따라 패칭 방식을 구분합니다. 핵심 기준은 "사용자 상호작용에 따라 데이터가 다시 바뀌는가?" 입니다.

| 구분             | 서버 컴포넌트 (Server Component)         | TanStack Query                                   |
| ---------------- | --------------------------------------- | ------------------------------------------------ |
| 데이터 성격      | 요청 시점에 한 번 가져와 렌더하면 끝     | 클라이언트에서 반복적으로 변하는 데이터          |
| 대표 사례        | 상세 화면, 목록 첫 페이지, 대시보드 초기값 | 무한 스크롤, 검색/필터/정렬, 폴링, mutation 갱신 |
| 위치             | `app/` 의 서버 컴포넌트                  | 클라이언트 컴포넌트 + ViewModel(`use-*.ts`)      |

**TanStack Query를 선택하는 신호** (하나라도 해당되면 TanStack Query)
- mutation 이후 자동 갱신(`invalidateQueries`)이 필요하다
- 여러 컴포넌트가 같은 데이터를 공유하며 캐시/무효화가 필요하다
- 로딩/에러/리페치 상태를 UI에서 세밀하게 다뤄야 한다
- 폴링, window focus refetch, 백그라운드 갱신이 필요하다
- 데이터가 유저 토큰/세션 등 클라이언트 컨텍스트에 의존한다

**기본 전략: 둘은 경쟁이 아니라 계층입니다.**
- 초기 1회 데이터 = **서버 컴포넌트** (로딩 스피너 없는 첫 화면, SSR 이점)
- 그 이후의 생애주기(refetch, mutation, 무한 스크롤) = **TanStack Query**
- 서버 컴포넌트에서 `prefetchQuery` → `dehydrate`, 클라이언트의 `HydrationBoundary`로 캐시를 이어받아 처리합니다.

> ⚠️ 이 프로젝트는 변경된 버전의 Next.js(v16)를 사용합니다. prefetch/hydration API 시그니처와 서버 컴포넌트 규칙이 일반적인 Next.js와 다를 수 있으므로, 구현 전 `node_modules/next/dist/docs/`의 해당 가이드를 반드시 확인하세요.

ViewModel(`use-*.ts`)에서 비동기 상태를 다룰 때는 TanStack Query를 사용하며, View는 가공된 결과만 props로 받습니다(4번 규칙 준수).
